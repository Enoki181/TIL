# Tech Log

## LIFFアプリにおけるセッション管理

### 前提

HTTPは **ステートレス** なプロトコルであるため、通常サーバーは「同じユーザーかどうか」を覚えていない。  
Railsではこの問題を **Session + Cookie** によって解決する。

---
### LIFFアプリのログイン構造

LIFFアプリでは、実際にはログインが3段階存在する。
```
① LINEログイン（LINEアプリ）
② LIFFログイン（LINE → Webアプリ許可）
③ Railsセッション（Webアプリ側ログイン）
```
役割はそれぞれ異なる。
| ログイン       | 役割                        |
| ---------- | ------------------------- |
| LINEログイン   | ユーザーがLINEアカウントにログインしている状態 |
| LIFFログイン   | LINEユーザー情報をWebアプリに渡す許可    |
| Railsセッション | アプリ内ユーザーとしてログイン状態を維持      |

---
### LIFFアプリのログインフロー

一般的な流れは以下。
```
ユーザー
↓
LINEアプリでLIFFを開く
↓
liff.init()
↓
liff.isLoggedIn()でログイン確認
↓
LINE userId取得
↓
RailsへPOST
↓
User.find_or_create_by(line_user_id)
↓
session[:user_id] = user.id
↓
Cookie保存
```
これによりRails側で **ログイン状態（current_user）** が判定できる。

---

### Railsセッションの仕組み

ログイン成功時に以下を実行する。

```ruby
session[:user_id] = user.id
```

Railsはこのsessionデータを暗号化し、Cookieとしてブラウザに保存する。

次のリクエストではブラウザが自動的にCookieを送信する。
```
GET /dashboard
Cookie: _app_session=xxxxx
```

RailsはCookieを復号し、下記のようにユーザーを特定する。

```
session[:user_id]
↓
User.find
↓
current_user
```

---
### Cookie削除時の挙動

Cookieを削除すると下記のようになり、ログイン状態は失われる。

```
Cookieなし
↓
session空
↓
current_user = nil
```
ただし、usersテーブルなどのDBデータは残るため、再ログインは可能。

---
### セキュリティ対策（Session Fixation）

ログイン時にはセッション固定攻撃を防ぐため`reset_session`を実行することが推奨される。

```
ログイン前 session
↓
reset_session
↓
新しいsession
↓
session[:user_id]
```

これにより攻撃者が既存セッションを利用することを防ぐ。

---
### LIFF + Rails セッションの役割

```
LINE userId
↓
Rails User作成
↓
session[:user_id]
↓
Cookie
↓
current_user
```

つまり下記のような構造になる。

```
LINEユーザー
↓
Railsユーザーに変換
↓
セッションでログイン状態維持
```
